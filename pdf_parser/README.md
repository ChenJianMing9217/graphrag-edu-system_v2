# IEP PDF to Neo4j Knowledge Graph Pipeline

---

## 一、 程式功能說明

本系統由三支核心程式組成：

### 1. `pdf_parser.py` (解析核心)
- **職責**：將 PDF 原始檔案轉換為結構化的 JSON 格式。
- **關鍵技術**：
    - 使用 `pdfplumber` 進行版面分析。
    - **Checkbox 偵測**：自動識別圖形勾選框（■/□）與文字符號（☑/☐），判斷評估項目是否選中。
    - **視覺化邏輯**：利用坐標關係將側邊欄標籤（主訴、診斷等）與其對應的內容進行配對。
    - **效能優化**：預編譯正則表達式，減少重複解析開銷。

### 2. `neo4j_importer.py` (資料整合)
- **職責**：清除舊資料，並將 JSON 批量匯入 Neo4j 資料庫。
- **關鍵技術**：
    - **語義化標註**：為項目加上 `(建議加強)`、`(目前有此疑慮)` 等前綴，讓 LLM 能直接讀懂勾選意義。
    - **批量寫入**：利用 Cypher `UNWIND` 語法，大幅提升資料庫寫入效率（比逐條插入快約 80%）。
    - **節點穩定性**：生成穩定的唯一 ID，確保多次匯入不會產生重複節點或斷開的孤島。

### 3. `pdf_processor_main.py` (自動化調度)
- **職責**：主控台入口與模組化整合。
- **功能**：
    - **全流程自動化**：依序啟動「PDF 解析」->「JSON 封存備份」->「Neo4j 匯入」。
    - **封存管理**：在 `IEP_TO_JSON/` 目錄下自動生成時間戳備份檔案，確保原始資料可追溯。
    - **模組介面**：提供 `IEPPipeline` 類別，方便其他 AI 應用程式直接引用。

## 二、 知識圖譜結構說明 (Knowledge Graph Schema)

圖譜採用「中心放射狀」與「層級分明」的設計，旨在模擬醫師/治療師的思辨邏輯，方便 LLM 進行語義檢索。

### 1. 核心層級關係
- **`Report` 節點**：圖譜的根，儲存報告標題、醫院名稱。
- **`Meta` 節點**：儲存患者基本資料（姓名、年齡、出生日期、病歷號）。
- **`Summary` 節點**：摘要中心，連接所有全院層級的結論。
- **`Domain` 節點 (發展領域)**：例如：`知覺動作`、`口語溝通`、`社會情緒`。
- **`Subdomain` 節點 (細分科目)**：例如：`知覺動作` 下的 `粗大動作` 或 `精細動作`。

### 2. 功能分類節點 (`CategoryHub`)
為了避免節點爆炸，我們在每一層下方建立「Hub (中心)」節點，將相同屬性的資訊歸類：
- **`主訴摘要`**、**`診斷摘要`**：連結具體的診斷結果。
- **`評估工具`**：列出該次評估使用的量表。
- **`行為觀察`**：記錄治療師在診間的質性觀察。
- **`訓練方向`**：記錄未來的發展計畫 (附帶語義標籤)。

### 3. 語義關係 (Relationships)
- `(Report)-[:HAS_DOMAIN]->(Domain)`：層級歸屬。
- `(Subdomain)-[:TRAINED_BY]->(TrainingDirection)`：指引未來訓練。
- `(CategoryHub)-[:OBSERVED]->(Observation)`：連結具體觀察。
- `(Subdomain)-[:EVALUATED_STATUS]->(StatusNode)`：記錄當前發展狀態（如：遲緩）。

### 4. 針對 LLM 的優化：Node Text 裝飾
圖譜中的資訊節點（如 TrainingDirection）在 `text` 欄位中會包含語義資訊：
- 例如：`[text: "(建議加強) 軀幹控制"]`。
- 這讓解釋機器人在抓取到此節點時，能直接說出：「針對軀幹控制，報告中建議我們需要加強...」而不需要額外的資料對齊邏輯。
